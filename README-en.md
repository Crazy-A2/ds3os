# What is DSOS?
DSOS is an open-source implementation of the game servers for Dark Souls 3 and Dark Souls 2: Scholar of the First Sin (SOTFS).

This project mainly aims to provide a relatively isolated online environment, so players can reduce ban risks when using mods, or play privately with friends without dealing with cheaters, invasions, and other problems common in public online play.

If you run into issues, you can join the Discord for technical support:
https://discord.gg/pBmquc9Jkj

# Component Overview
This repository currently consists of the following major parts:

- `Loader`: Windows GUI launcher used to fetch server lists, start the game, and connect to custom servers
- `Injector`: Windows-only injection module used to redirect the game to custom servers
- `Server`: Main server program handling login, authentication, gameplay logic, and the WebUI service
<!-- - `MasterServer`: Node.js API service for publishing and listing active servers -->
- `WebUI`: Static assets for the server management page, distributed with the server

If this is your first time looking at this repository, you can think of it as:

1. Players usually use `Loader` to join servers
2. Server hosts usually run `Server.exe` to host their own server
<!-- 3. You only need `MasterServer` when maintaining a fork or public server list -->

# How do I use it?
After building, outputs are generated by platform, architecture, and mode, for example:

- `bin/x64_release/`
- `bin/x64_debug/`
- `bin/x86_release/`

In a common Windows x64 workflow, `xmake build-server` and `xmake build-all` also organize outputs into:

- `bin/<arch>_<mode>/Loader/`
- `bin/<arch>_<mode>/Server/`

The most commonly used parts are `Loader/` and `Server/`.

- `Loader`: starts the game and connects it to a custom server
- `Server`: starts your own dedicated server

## Player Quick Start
1. Prepare the launcher in the `Loader` directory.
2. Run Loader as administrator.
3. In Loader, select the game to launch, refresh the server list, then join your target server.
4. If the server has password protection enabled, enter the password before connecting.

## Host Quick Start
1. Run `Server.exe` in the `Server` directory.
2. On first startup, it generates `Saved/default/config.json`.
3. Edit the config and restart the server to apply changes.
4. If you want public internet access, also check port forwarding, firewall rules, and WAN/LAN address settings.

If you want password protection, set a password in `Saved/default/config.json`. Players will then need to enter that password before connecting.

**Note:** The **Steam client** must be installed on the same machine when running `Server.exe` (login is not required). Otherwise `Server.exe` may fail to initialize properly.

# Will this get my retail account banned?
DSOS uses separate save files.

As long as you **do not copy DSOS saves back into your retail saves**, you should generally be fine.

# FAQ
## How do I switch between Dark Souls 3 and Dark Souls 2?
After the server runs for the first time, it creates `Saved/default/config.json`.

You can modify the `GameType` parameter between `DarkSouls2` and `DarkSouls3` to choose which game type the current server hosts.

## Why can't I see my saves?
DSOS uses its own separate save files to avoid affecting retail saves.

If you want to import your retail saves into DSOS, click the settings (gear) icon at the bottom of Loader, then use the "Copy retail saves" option.

For safety reasons, the project **does not provide** an automatic way to copy DSOS saves back into retail saves.

If you **really know what you are doing**, you can manually locate the save directory and rename the `.ds3os` extension to `.sl2`.

## The game starts, but it cannot connect to the server. What should I check?
Common causes include the following.

First, verify whether Loader is running **as administrator**. The launcher needs to patch game memory so the game connects to the custom server, and this usually requires admin privileges.

If you host the server yourself and that still does not solve it, continue checking:

1. Whether your router forwards these ports for both TCP and UDP: `50000`, `50010`, `50050`, `50020`
2. Whether Windows Defender Firewall allows the server executable
3. Whether network settings in `Saved/config.json` are correct

The most critical fields are:

- `ServerHostname`: should be your public/WAN IP
- `ServerPrivateHostname`: should be your LAN IP

If you are using LAN simulation tools (for example, Hamachi), set them to the corresponding virtual LAN IPs.

## What do fields in the config file mean?
Configuration field documentation is primarily maintained in source code here:
https://github.com/Crazy-A2/ds3os/blob/main/Source/Server/Config/RuntimeConfig.h

Common fields quick reference:

| Field | Description |
| --- | --- |
| `GameType` | Server game type, `DarkSouls3` or `DarkSouls2` |
| `ServerHostname` | Public/WAN IP used by internet players |
| `ServerPrivateHostname` | LAN IP used by local players |
| `Password` | Server password; empty means no password required |
| `Advertise` | Whether to publish this server to MasterServer |
| `ServerName` | Display name of the server |
| `Announcement` | Announcement text shown when players join |

# How do I build it?
This fork currently focuses on **Windows + XMake**, and the build entry is `xmake.lua` in the repository root.

## Requirements
- [Visual Studio BuildTools](https://aka.ms/vs/stable/vs_BuildTools.exe) or [Visual Studio](https://visualstudio.microsoft.com/vs/)
- Windows SDK
- [.NET 10 SDK](https://dotnet.microsoft.com/download)（required only when building Loader）
- [XMake](https://xmake.io/guide/quick-start.html)

## Build Steps

### 1. Configure build environment
Run this once before the first build:

```bash
xmake config --plat=windows --arch=x64 --mode=release --toolchain=msvc
```

### 2. One-command build (recommended)
```bash
xmake build-all
```

This command performs, in order: configure MSVC environment → build Server → build Injector → build Loader → organize output directories → copy runtime assets.

### 3. Build in steps
If you only need specific parts:

```bash
# Build server (on Windows x64, also builds Injector and Loader)
xmake build-server

# Build Loader only (Windows x64 only)
xmake build-loader

# Build Server executable only
xmake build Server

# Build Injector DLL only
xmake build Injector
```

### 4. Copy runtime assets
If you only want to copy WebUI, Steam runtime libraries, and other assets into output directories without recompiling:

```bash
xmake install-all
```

## Common XMake Tasks

| Command | Actual behavior |
| --- | --- |
| `xmake build-all` | Calls `build-server`, then `install-all` |
| `xmake build-server` | Builds Server; on Windows x64 it also builds Injector and Loader, then organizes output |
| `xmake build-loader` | Windows x64 only; builds Injector and Loader |
| `xmake install-all` | Copies `Source/WebUI`, Steam runtime libs, and dynamic libs under `lib/` to output dirs |
| `xmake clean-bin-libs` | Cleans `.lib` / `.pdb` files under `bin/` |

## Output directory layout
Build artifacts are output to `bin/<arch>_<mode>/` by default, for example:

```text
bin/
├── x64_release/
│   ├── Server/        ← Server.exe, WebUI, steam_api64.dll, etc.
│   └── Loader/        ← Loader.exe, Injector.dll, etc.
├── x64_debug/
└── x86_release/
```

The `Server/` directory also contains static assets from `Source/WebUI`, and Server serves WebUI directly from there after startup.

## OpenSSL notes
By default, prebuilt OpenSSL libraries from `lib/` are linked. If you need to build OpenSSL from source, add:

```bash
xmake config ... --bundled_openssl=y
```

## MasterServer (optional)
MasterServer is an independent Node.js service. You only need to run it if you want to maintain your own server list.

```bash
cd Source/MasterServer
npm install
npm run start        # production mode
npm run start:dev    # development mode (nodemon auto-restart)
```

You can also use a Docker image (see Docker notes below).

# Repository Layout
```text
/
├── xmake.lua              Build entry, defines all C++ build targets
├── plugins/               Custom XMake tasks (build-all, build-server, etc.)
├── Protobuf/              Protobuf definitions for server network communication
├── Resources/             Shared build/package resources, such as icons and documents
├── Source/                All project source code
│   ├── Injector/          DLL injected into the game (Windows only), redirects online target
│   ├── Loader/            WinForms launcher (Windows x64 only, .NET 10)
│   ├── MasterServer/      Server registry, Node.js API service
│   ├── Server/            Main server framework (login, auth, gameplay logic, WebUI)
│   ├── Server.DarkSouls3/ Dark Souls 3 specific implementation
│   ├── Server.DarkSouls2/ Dark Souls 2 specific implementation (experimental)
│   ├── Shared/            Core library shared by server and injector
│   ├── ThirdParty/        Third-party source libraries (openssl, curl, sqlite, protobuf, etc.)
│   └── WebUI/             Static assets for the server management page, distributed with Server after build
├── Tools/                 Helper scripts (generate VS project, protobuf, packaging, etc.)
└── lib/                   Prebuilt OpenSSL libraries (default linking method)
```

# How can I contribute?
You can check Issues, or contact maintainers directly to learn which areas currently need the most help.

High-value areas currently include:

- Improving server calls that are still stubs or only return placeholder data
- Reverse-engineering the exact response formats those interfaces should return
- Identifying more unknown protobuf field meanings

These efforts directly improve the completeness of the server implementation.
